# Содержательные имена
Имена встречают повсюду. То, что делается так часто, должно делаться хорошо. В главе приведены простые правила по поводу именования.

## Имена должны передавать намерения программиста
Имя переменной, функции или класса должно ясно говорить, зачем она существует, что она делает и как используется. Хорошее имя не требует комментария.

```java
// Плохо:
int d; // Прошедшее время


// Хорошо:
int elapsedTimeInDays;
int daysSinceCreation;
int daysSinceModification;
int fileAgeInDays;
```

*Очевидные имена существенно повышают читаемость.*

## Избегайте дезинформации
Избегайте ложных ассоциаций, искажающих восприятие кода. Избегайте слов со скрытым смыслом, аббревиатур и конкретных терминов не по назначению:

Переменную `accountList` лучше переименовать в `accountGroup`, `bunchOfAccounts` или просто в `accounts`. Даже не смотря на то, что "List" действительно может указывать на структуру данных.

Остерегайтесь крайне сходных имен, которые различаются не столь очевидно.

Учитывайте написание шрифтов и будьте осторожны с, например, `l`/`1` и `O`/`0`.

## Используйте осмысленные различия
Пишите код, ориентируясь на людей, а не на компиляторы.

Если функция перемещает элементы массивов из `a1` в `a2` -- это плохие имена параметров. Гораздо лучше: `source` и `destination`.

Если нет жестких именных схем, то
* `moneyAmount` не отличается от `money`
* `customerInfo` от `customer`
* `accountData` от `account`
* `theMessage` от `message`

Читатель кода должен понимать различие имен и какое из них ему требуется использовать, чтобы решить свою задачу.

## Используйте удобнопроизносимые имена
```java
// Плохо:
class DtaRcrd102 {
  private Date genymdhms;
  private Date modymdhms;
  private final String pszqint = "102";
  /* ... */
}

// Примечание: genymdhms это дата генерирования,
// год, месяц, день, час, минута и секунда


// Хорошо:
class Customer {
  private Date generationTimestamp;
  private Date modificationTimestamp;
  private final String recordId = "102";
  /* ... */
}
```

## Выбирайте имена, удобные для поиска
Мнение автора: *однобуквенные имена могут использоваться ТОЛЬКО для локальных переменных в коротких именах*.

Избегайте "магических констант".

## Избегайте схем кодирования имен
Кодирование информации о типе или области видимости в именах только создает новые хлопоты по расшифровке. Также, как правило, кодированные имена плохо произносятся и в них легко допустить опечатку.

### Венгерская запись
Венгерская запись была необходима в прошлом, когда компиляторы не поддерживали проверку типов. Современные языки лишены этого недостатка, поэтому венгерская запись сегодня лишь усложняет изменение имени или типа сущности.

`PhoneNumber phoneString;`

### Префиксы членов классов
Префиксы "m_" -- тоже пережиток прошлого. Классы и функции должны быть достаточно компактными, чтобы обходиться без них.

```java
// Плохо:
public class Part {
  private String m_dsc;   // текстовое описание
  void setName(String name) {
    m_dsc = name;
  }
}


// Хорошо:
public class Part {
  String description;
  void setDescription(String description) {
    this.description = description;
  }
}
```

### Интерфейся и реализации
Задача: объявить абстрактную фабрику для создания геом. фигур. Названия `IShapeFactory` и `ShapeFactory` -- отвергаются автором. Префикс `I` отвлекает от кода или даже передает лишнюю информацию. Имена `ShapeFactoryImp` или `CShapeFactory`, по мнению автора, кодируют информацию об интерфейсе лучше.

## Избегайте мысленных преобразований
*Ясность превыше всего*.

Переменная `r` не должна содержать URL-адрес с удаленным хостом и схемой.

Счетчики циклов можно назвать `i`, `j` или `k`, если их область видимости мала.

### Имена классов
* это существительное и их комбинации
`Customer, WikiPage, Account, AddressParser, ...`

* старайтесь не использовать
`Manager, Processor, Data, Info, ...`

* имя не должно быть глаголом

### Имена методов
* это глагол или глагольное словосочетание
`postPayment, deletePage, save, ...`

* Методы чтения/записи и предикаты образуются из префикса `get, set, is` и значения (стандарт javabean)
```java
string name = employee.getName();
customer.setNam("mike");
if (paycheck.isPosted()) {
  ...
}
```

### Избегайте остроумия
Отдавайте предпочтение ясности перед развлекательной ценностью.

### Выберите одно слово для каждой концепции
Единый лексикон помогает избежать лишней путаницы. Выберите одно слово для представления одной абстрактной концепции и придерживайтесь его.

Использование `fetch, retrieve, get` для эквивалентных методов разных классов -- плохо.

Аналогичным образом, термины `controller, manage, driver` создают путаницу. Например, неочевидно принципиальное отличие `DeviceManager` от `ProtocolController`.

### Воздержитесь от каламбуров
Старайтесь не использовать одно слово в двух смыслах (идеях).

Если есть множество методов `add` и ваш новый метод не похож на них семантически, то лучше дать ему имя `insert` или `append`.

### Используйте имена из пространства решения
Код будут читать программисты, используйте это знание: не стесняйтесь использовать общепринятые термины из информатики, названия алгоритмов, паттернов и т.д.

`AccountVisitor` говорит об использованном паттерне, а `JobQueue` об очереди как структуре.

### Используйте имена из пространства задачи
Если для описания чего-либо вы не можете подобрать программистический термин, возьмите его из предметной области.

### Добавьте содержательный контекст
Есть немного имен, которые содержательны (грубо говоря, независимы) сами по себе. Все остальные имена следует помещать в определенный контекст для чтения кода, заключая их в классы/функции/пространство имен. В крайнем случае, можно дать имени префикс.

Пример: переменным с именами `firstName, lastName, street, houseNumber, city, state, zipcode` соответствует контекст "адрес".
Если встретить в коде просто `state`, то можно не сразу догадаться, что он относиться к адресу.

Правильным решением будет вынести их в отдельный класс `Address`.
Другим решением будет добавить им префикс: `addrFirstName, addrLastName...`

### Не добавляйте избыточный контекст
Не лучшая идея добавлять название приложения как префикс в каждый класс.

Короткие имена обычно лучше длинных ,если только их смысл понятен читателю кода. *Не включайте в имя больше контекста, чем необходимо*.

`accountAddress, customerAddress` -- неплохие имена для переменных, плохие имена для класса. `Address` подходит лучше. Для подчеркивания различных адресов, следует именовать их как `PostalAddress, MAC, URI`.
